# 商品视频预览设计

## 背景

当前项目的商品模型仅支持图片字段 `image`。首页商品卡片展示商品图片，商品管理页支持上传和维护图片，但不支持视频。用户希望在不破坏现有图片展示链路的前提下，给商品增加可选视频，并在首页通过点击商品图片弹窗播放竖屏短视频。

## 目标

本次设计的目标如下：

1. 商品支持绑定一个可选视频资源。
2. 首页商品卡片在存在视频时显示播放角标。
3. 用户点击首页商品图片后，以弹窗形式播放竖屏短视频。
4. 商品管理页支持上传、替换、清除商品视频。
5. 保持现有图片封面逻辑不变，不把首页列表改成直接播放视频。

## 非目标

本次不包含以下内容：

1. 不新增独立的商品详情页。
2. 不支持一个商品绑定多个视频。
3. 不把视频展示扩展到购物清单、销售记录或个人页。
4. 不引入媒体资源数组 `mediaList` 之类的通用媒体模型。
5. 不做历史数据迁移脚本，旧数据按无视频处理。

## 方案比较

### 方案 A：图片封面 + 可选视频 + 首页弹窗播放

这是推荐方案。

- `image` 继续作为商品封面图。
- 新增 `video` 可选字段，保存视频资源。
- 首页继续展示图片列表，仅在点击图片时按需拉取视频地址并弹窗播放。

优点：

- 对现有页面破坏最小。
- 列表性能稳定，避免在滚动列表中挂载视频组件。
- 首页交互直接符合“点图即看”的目标。

缺点：

- 需要补充视频上传能力、视频弹窗组件和首页状态管理。

### 方案 B：点击图片跳转独立视频页

- 首页点图后跳转到独立页面播放视频。

优点：

- 播放器状态更简单。
- 页面隔离明确。

缺点：

- 与用户要的“弹窗播放”不一致。
- 多一次页面跳转，打断浏览节奏。

### 方案 C：统一媒体数组

- 商品改为维护图文视频资源数组，首页从中推导封面和视频。

优点：

- 长期扩展性最好。

缺点：

- 对当前项目明显过度设计。
- 会连带改动类型、云函数、首页、商品管理、清单等多个模块。

## 结论

采用方案 A：图片封面 + 可选视频 + 首页弹窗播放。

## 数据模型

### Fruit 类型

在现有 `Fruit` 上新增可选视频字段：

```ts
export interface Fruit {
  _id?: string;
  id: number;
  category: string;
  name: string;
  desc: string;
  price: number;
  costPrice: number;
  image: string;
  video?: string;
}
```

设计约束：

1. `image` 的语义不变，仍然是首页和商品管理列表的封面图。
2. `video` 为可选字段；无视频商品允许缺失或为空字符串。
3. `video` 存储微信云存储 `fileID`，不直接存公网视频地址。

### 云数据库记录

`fruits` 集合中的每条商品记录新增 `video` 字段，更新与新增时一并读写。旧记录缺失 `video` 时视为无视频，不需要单独迁移。

## 页面职责

### 商品管理页

职责：

1. 上传和维护商品图片。
2. 上传、替换、清除商品视频。
3. 保存商品基础信息、封面图和视频信息。

边界：

1. 商品管理页不负责首页播放逻辑。
2. 商品管理列表不直接内嵌视频播放，只提供上传状态和预览入口。

### 首页

职责：

1. 展示商品封面图和基础信息。
2. 根据 `video` 是否存在，决定是否显示播放角标。
3. 管理当前选中视频和弹窗显隐状态。

边界：

1. 首页不负责视频上传。
2. 首页不直接维护视频文件地址，只在点击时按需获取临时地址。

### 视频弹窗组件

建议抽成独立组件，例如 `VideoPreviewModal`。

输入建议：

- `visible: boolean`
- `title: string`
- `videoUrl: string`
- `onClose: () => void`

职责：

1. 渲染全屏遮罩与弹窗容器。
2. 播放竖屏短视频。
3. 提供关闭、重播等基础交互。
4. 在关闭或卸载时停止播放并清理状态。

## 首页交互设计

### 封面态

1. 首页商品卡片仍显示 `image`。
2. 只有 `video` 存在时，图片右下角显示播放角标。
3. 无视频商品不显示角标。

### 点击行为

1. 用户点击商品图片。
2. 如果商品无 `video`，不触发视频弹窗。
3. 如果商品有 `video`：
   - 首页进入加载态。
   - 通过 `fileID` 获取视频临时地址。
   - 获取成功后打开视频弹窗并自动播放。

### 播放态

1. 弹窗为居中模态层，背景半透明遮罩。
2. 视频容器按竖屏短视频布局展示。
3. 标题显示商品名。
4. 支持以下关闭方式：
   - 点击右上角关闭按钮。
   - 点击遮罩空白区。

### 关闭行为

关闭弹窗时：

1. 停止视频播放。
2. 清空当前选中视频 URL。
3. 保持首页原滚动位置，不跳转、不刷新列表。

## 异常与兜底

1. `video` 为空：不显示角标，不弹视频。
2. 获取临时 URL 失败：提示“视频加载失败”，不打开空白弹窗。
3. 视频资源失效：弹窗内显示失败占位态，允许关闭。
4. 用户连续点击多个商品：只保留最后一次点击对应的视频。
5. 页面切走、组件卸载或弹窗关闭：播放器必须停止播放并释放状态。

## 上传与存储设计

### 商品管理页上传流程

建议使用 `Taro.chooseMedia`，限制为视频：

1. 点击“选择视频”。
2. 打开相册或相机，选择单个视频。
3. 客户端做基础校验：
   - 仅允许单个视频。
   - 最大时长建议 60 秒。
4. 调用云存储上传接口。
5. 上传成功后保存返回的 `fileID` 到 `editData.video`。

### 云存储目录

建议的视频路径格式：

```ts
fruit-videos/${Date.now()}-${random}.mp4
```

### 视频预览 URL

数据库中保存 `fileID`，首页播放前再调用 `Taro.cloud.getTempFileURL` 获取临时可访问 `https` 地址。

这样做的原因：

1. 和现有图片上传链路保持一致。
2. 避免数据库存放不稳定的临时 URL。
3. 降低外部域名和播放器兼容问题。

## 测试范围

### 功能验证

1. 无视频商品：首页不显示角标。
2. 有视频商品：首页显示角标。
3. 点击有视频商品图片后，能正常弹窗并播放竖屏视频。
4. 点击无视频商品图片时，不会出现错误弹窗。
5. 商品管理页上传视频后，重新加载商品列表仍能播放。
6. 替换视频后，首页播放最新视频。
7. 清除视频后，首页角标消失。

### 异常验证

1. 上传超时或失败时，界面给出明确提示。
2. 获取临时 URL 失败时，弹出失败提示且不进入坏状态。
3. 连续点击不同商品视频时，不出现旧视频残留。
4. 关闭弹窗后，视频停止播放。

## 风险与限制

1. 视频资源比图片大，必须限制时长和数量，否则首页体验会变差。
2. 如果未来要支持多媒体轮播，本设计需要升级为更通用的媒体模型。
3. 数据库存 `fileID` 后，播放链路依赖云存储和临时链接获取成功。

按以下顺序实施：

4. 实现首页卡片角标和点击事件。
5. 新增视频弹窗组件并接入首页。
